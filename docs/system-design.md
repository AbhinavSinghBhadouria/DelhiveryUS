# System Design Write-Up

The Last-Mile Delivery Tracker is designed around three roles: customer, admin, and delivery agent. Customers create orders and track delivery progress, admins configure operational rules and manage orders, and agents execute deliveries through controlled status updates. The backend exposes role-protected APIs, while the frontend provides separate workflows for each role.

The rate calculation engine is fully configuration-driven. Admins maintain zones, area or pincode mappings, B2B and B2C rate cards, and COD surcharges. During order quotation, the system detects the pickup and drop zones from the entered address metadata. It calculates volumetric weight using `length * breadth * height / 5000`, compares it with actual weight, and bills on the higher value. The engine then selects the correct rate card by order type, pickup zone, drop zone, and whether the movement is intra-zone or inter-zone. If payment type is COD, the configured COD surcharge for the order type is added. The customer sees the final charge before confirming the order, and the calculated values are stored with the order for auditability.

Zone detection uses admin-managed area and pincode mappings. Each zone can contain multiple area records, and each area record can include city, state, and pincode data. At order creation time, pickup and drop pincodes are matched against this mapping. If a zone cannot be identified, the system blocks order creation with a clear validation message instead of guessing. This keeps pricing predictable and forces admins to configure serviceable areas explicitly.

Agent assignment supports both manual and automatic flows. In manual assignment, an admin chooses an available agent. In auto-assignment, the system first searches for available agents in the pickup zone. If agents have latitude and longitude data, it ranks them by distance from the pickup location. If exact location is unavailable, it falls back to same-zone availability and active order count. The selected agent is linked to the order, and an assignment event is added to the tracking history.

Order tracking is append-only. Every status transition creates a tracking event with old status, new status, actor, timestamp, and optional notes. Existing tracking rows are never edited or deleted, giving admins and customers a reliable delivery timeline. Agents can move orders through picked up, in transit, out for delivery, delivered, and failed states. Admins can override statuses when required.

Failed delivery is handled as a separate recovery flow. When an agent marks an order as failed, the customer is notified by email and SMS, and the order remains visible for rescheduling. The customer submits a new delivery date, the system records a reschedule request, updates the order status, and triggers reassignment manually or automatically. Notifications are generated for each status change and stored in a notification log so delivery communication can be audited.
